# FVM Roles & Interpretation Guide

## Purpose

This reference defines the role hierarchy and interpretation rules for the Functional Verification Matrix (FVM). Scouts use this to understand expected access patterns; validators use this to classify failures.

## Role Hierarchy

Visibility tiers are ordered from least to most restrictive:

```
public < authenticated < team_member < team_manager < team_owner < superadmin
```

### Tier × Role Matrix

| Tier | unauthenticated | viewer | member | manager | owner | superadmin |
|------|:-:|:-:|:-:|:-:|:-:|:-:|
| public | ✅ | ✅ | ✅ | ✅ | ✅ | ✅ |
| authenticated | ❌ | ✅ | ✅ | ✅ | ✅ | ✅ |
| team_member | ❌ | ✅ | ✅ | ✅ | ✅ | ✅ |
| team_manager | ❌ | ❌ | ❌ | ✅ | ✅ | ✅ |
| team_owner | ❌ | ❌ | ❌ | ❌ | ✅ | ✅ |
| superadmin | ❌ | ❌ | ❌ | ❌ | ❌ | ✅ |

- ✅ = Expected HTTP 200 (or 2xx/404 for valid resource access)
- ❌ = Expected HTTP 401 (unauthenticated) or 403 (authenticated but insufficient role)

### Role Definitions

| Role | System Role | Team Role | Scope |
|------|------------|-----------|-------|
| superadmin | superadmin | — | Full system access, all tiers pass |
| owner | user | owner | Team owner, manages team settings and providers |
| manager | user | manager | Team admin, manages channels, secrets, members |
| member | user | member | Team member, creates workflows, uses workspace |
| viewer | user | viewer | Read-only team access, can view but not modify |
| unauthenticated | — | — | No session, only public pages accessible |

### Special Cases

- **Auth disabled**: When authentication is not enabled on the server, all tiers except `public` are treated as `public` (single-user mode). FVM validation skips auth-based rows.
- **superadmin bypass**: superadmin passes ALL tier checks, including `team_*` tiers, regardless of team membership.
- **view vs manage**: Each route has two tiers. `view` controls page access (GET/read). `manage` controls mutations (POST/PUT/PATCH/DELETE). FVM uses `view` tier for GET endpoints and `manage` tier for mutation endpoints.

## FVM Failure Classification

| Code | Meaning | Severity | Action |
|------|---------|----------|--------|
| AUTH_LEAK | Role denied by policy, but server returned 2xx | 🔴 Critical | Security fix required — add auth guard |
| FALSE_DENY | Role allowed by policy, but server returned 403/401 | 🟡 High | Bug fix — check middleware or handler auth logic |
| ENDPOINT_MISSING | Server returned 404 for defined endpoint | 🟠 Medium | Check route registration or endpoint path |
| PARAM_ERROR | Server returned 400/422 for valid endpoint | 🟢 Low | Check request body schema, may be test data issue |

### Interpreting Dynamic Endpoints

Endpoints with path parameters (e.g., `/api/admin/teams/:id`) are called with a placeholder ID. Expected behaviors:
- **200**: Endpoint works (resource may or may not exist)
- **404**: Endpoint works, resource not found (acceptable for `:id` routes)
- **403/401**: Access denied (expected for unauthorized roles)
- **400**: Invalid parameter format (may be acceptable for placeholder values)

## Integration with RTM

FVM complements RTM — they serve different verification layers:

| Aspect | RTM | FVM |
|--------|-----|-----|
| Level | Source code | HTTP runtime |
| Verifies | Code exists, tests pass | API contracts work, auth enforced |
| Generated by | Scout (code_map, grep) | fvm_generate (static analysis) |
| Validated by | Implementer (test runner) | fvm_validate (HTTP requests) |
| Failure type | Implementation gap | Integration/security gap |

Scout should generate both RTM and FVM when frontend routes are in scope.
