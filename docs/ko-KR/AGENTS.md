# quorum AI 에이전트 가이드

> quorum이 설치된 프로젝트에서 AI 에이전트가 따라야 할 가이드.

## 역할 체인

| 역할 | 책임 | 모델 | 턴 |
|------|------|------|-----|
| **플래너** | 트랙 정의 + 실행 계획 | Opus | 30 |
| **wb-parser** | WB 파일 파싱 → 요구사항 테이블 | Haiku | — |
| **rtm-scanner** | 코드 스캔 → Forward/Backward RTM 행 | Haiku | — |
| **스카우트** | RTM 갭 분석 + 교차 트랙 이슈 (Phase 5-8) | Sonnet | 30 |
| **디자이너** | 설계 문서 (Spec, Blueprint, Domain Model, Architecture) | Sonnet | — |
| **fde-analyst** | 요구사항별 실패 모드 체크리스트 | Haiku | — |
| **오케스트레이터** | 작업 분배 → 추적 → 게이트 체인 → 머지 | 메인 세션 | — |
| **구현자** | 워크트리에서 구현 + 테스트 + 증거 제출 | Sonnet | 30 |
| **self-checker** | 결정론적 CQ/T/lint/scope 검사 (구현 후) | Haiku | 15 |
| **fixer** | 감사 거절에 대한 표적 수정 (전면 재작성 아님) | Sonnet | 15 |
| **gap-detector** | RTM 기반 교차 트랙 갭 분석 | Haiku | — |
| **전문가** | 도메인 리뷰 (perf, a11y, security, ...) — confidence ≥ 0.8 | 도메인별 | 15 |
| **감사자** | 독립 검증 → 승인/거절 판정 | GPT/Codex | — |

## 역할 디스패치 파이프라인

```
Phase 1-4: 플래너 (Opus)
  └→ wb-parser (Haiku) → 요구사항 테이블
  └→ 디자이너 (Sonnet) → 설계 문서
  └→ fde-analyst (Haiku) → 실패 모드 체크리스트

Phase 5-8: RTM + 분석
  └→ rtm-scanner (Haiku) → Forward/Backward RTM 행
  └→ 스카우트 (Sonnet) → 갭 분석 + 교차 트랙 보고서

Phase N: Wave 실행 (WB별)
  └→ 구현자 (Sonnet) → 코드 + 테스트 + 증거
  └→ self-checker (Haiku) → CQ/T/lint/scope (결정론적만)
       ├→ PASS → 21-gate 체인 → 감사
       └→ FAIL → fixer (Sonnet) → 재검사
  └→ 감사자 → 판정
       ├→ 승인 → confluence 검사 → 회고 → 머지
       └→ 수정요청 → fixer (Sonnet) → 재감사
```

## 전체 사이클

```
플래너 → 설계 → RTM 생성 → 오케스트레이터 → 작업 분배
    ↓
┌─── 트랙 A (워크트리) ──────────────────────────────────────┐
│  구현자: 코드 + 테스트 + 증거 제출                          │
│  → self-checker (haiku): CQ/T/lint/scope — 결정론적         │
│       FAIL → fixer (sonnet): 표적 수정 → 재검사             │
│  → 21-gate 체인 (기계적)                                    │
│  → 감사 (트리거 평가: T1 스킵 / T2 단일 / T3 숙의)          │
│       T3: 옹호자 + 반대자 → 판사                            │
│  → 판정                                                    │
│       수정요청 → fixer → 재감사 (최대 3라운드)               │
│       승인 → confluence 검사 → 프로젝트 테스트               │
│  → 회고 (세션 게이트가 Bash/Agent 차단)                      │
│  → 머지 → squash → 단일 커밋 → 다음 트랙                    │
└──────────────────────────────────────────────────────────────┘
```

## 21-Gate 체인 (감사 전)

기계적 게이트 — 감사 전 전부 실행. LLM 미관여.

| # | 게이트 | 차단 여부 | 설명 |
|---|--------|----------|------|
| 1 | 회귀 감지 | 경고 | 파일 내용 50% 이상 교체 |
| 2 | 스텁/플레이스홀더 스캔 | 경고 | `TODO`, `FIXME`, 빈 구현체 |
| 3 | Blueprint 네이밍 린트 | **차단** | 네이밍 규칙은 법률 |
| 4 | 성능 안티패턴 | 경고 | O(n²) 루프, 동기 I/O, 무제한 쿼리 |
| 5 | 의존성 감사 | **차단** | Copyleft 라이선스 감지 |
| 6 | 파일 범위 적용 | 경고 | targetFiles 외 파일 변경 |
| 7 | Fitness 게이트 | **차단** | 점수 하락 >0.15 → 자동 거절 |
| 8 | 테스트 파일 생성 | 경고 | WB.verify = 테스트인데 테스트 파일 없음 |
| 9 | WB 제약 조건 | 경고 | 교차 WB 의존성 검증 |
| 10-21 | 도메인 스캔 | 다양 | perf, a11y, security, compat, ... |

## 수정안 투표

수정안 대상별 차등 임계값:

| 대상 | 임계값 | 근거 |
|------|--------|------|
| WB (작업 분해) | 50% 단순 다수 | 전술적 변경 |
| PRD (요구사항) | 66% 가중 다수 | 무엇을 만들지 변경 |
| Design (설계) | 66% 가중 다수 | 어떻게 만들지 변경 |
| Scope (범위) | 100% 만장일치 | 프로젝트 경계 변경 |

## 전문가 신뢰도 필터링

전문가 리뷰어는 신뢰도 기반 필터링 사용:

| 신뢰도 | 행동 |
|--------|------|
| 0.9–1.0 | 보고 (확실한 버그, 위반) |
| 0.8–0.89 | 보고 (매우 가능성 높은 이슈) |
| 0.5–0.79 | **보고 안 함** (자문용) |
| < 0.5 | 건너뜀 |

리뷰당 최대 10개 발견 사항, 심각도 × 신뢰도 기준 우선순위.

## 증거 패키지 형식

`audit_submit` 도구로 증거를 제출한다:

```markdown
## [trigger_tag] 작업 제목

### Claim
구체적으로 무엇을 했는지 서술.

### Changed Files
- `src/경로/파일.ts` — 변경 내용 설명

### Test Command
\`\`\`bash
npx vitest run tests/특정파일.test.ts
\`\`\`

### Test Result
\`\`\`
실제 터미널 출력을 그대로 붙여넣기.
\`\`\`

### Blast Radius (선택)
파일 → N 직접, M 전이 (비율%)

### Residual Risk
알려진 미해결 항목 또는 "없음".
```

## 절대 규칙

1. **`[trigger_tag]`만 사용** — 비표준 라벨 금지. 감사자만 `[agree_tag]` 적용.
2. **자기 승격 금지** — 자신의 코드에 `[agree_tag]`를 직접 적용할 수 없다.
3. **테스트 명령은 그대로 재실행 가능** — 감사자가 복사해서 실행한다.
4. **Changed Files는 실제 diff와 일치** — 불일치 시 `scope-mismatch` 거절.

## 플래너 문서 체계

트랙당 8개 문서. 각각 **별도 파일**, **단일 책임**.

| # | 문서 | 책임 |
|---|------|------|
| 1 | **PRD** | WHAT/WHY — 문제, 목표, 비목표, 리스크 |
| 2 | **Spec** | 인터페이스 — API, DDL, 환경변수, 에러 코드 |
| 3 | **Blueprint** | 구조 — 디렉토리, 네이밍 규칙 (= 법률) |
| 4 | **Domain Model** | 엔티티 — ER, 상태 머신, 불변식 |
| 5 | **실행 순서** | WHEN — 단계 의존성, 크리티컬 패스 |
| 6 | **테스트 전략** | HOW TO VERIFY — 유형, fixture, 커버리지 |
| 7 | **작업 분해** | HOW TO BUILD — 태스크별 Action/Verify/Done |
| 8 | **작업 카탈로그** | STATUS — ID, 제목, 크기, 상태, 의존성 |

**분리 원칙**: 각 문서는 서로 중복하지 않는다.
